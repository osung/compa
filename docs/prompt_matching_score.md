# 적합도 평가(매칭 점수) 프롬프트 (범용)

기준이 되는 문서와 후보 문서들 사이의 **실질 적합도를 0~100 점으로 산출**한다.
매칭 방향에 구애받지 않으며, 예로 다음을 모두 지원한다.

- 국가 R&D 과제 → 기업
- 기업 → 국가 R&D 과제
- 기업 → 특허
- 수요기술 → 국가 R&D 과제

이 문서는 특정 실행 파일이나 실행 하드웨어(NVIDIA GPU / Apple Silicon 등)에 의존하지 않는다. **Qwen 계열 LLM + 문서 문자열**만 있으면 그대로 재현할 수 있도록, 프롬프트 전문·파라미터·출력 파싱까지 자기완결적으로 기술한다.

## 개요

- **입력**
  - 기준측(source): 문서 텍스트(문자열) + 유형 라벨(`src_label`, 예: `"국가 R&D 과제"`, `"기업"`, `"수요기술"`)
  - 후보측(candidates): `{id, text}` 리스트(id=1..K) + 유형 라벨(`cand_label`, 예: `"기업"`, `"특허"`)
- **출력**: 각 후보에 대한 `{"results": [{"id", "score", "reason"}, ...]}` — `score`는 0~100 정수, `reason`은 짧은 한국어 근거
- **처리**: `System prompt` + (`기준 블록` + `후보 목록` + `가이드`)를 LLM에 전달 → 후보를 `LLM_BATCH`개씩 나눠 평가 → JSON 파싱 → `{id: (score, reason)}` 반환

## 모델 / 파라미터

- **모델**: Qwen3.6-35B-A3B 계열 (35B MoE, 활성 3B, 4bit 양자화)
  - 실행 백엔드·하드웨어와 무관하게 동일 프롬프트로 동작한다(같은 가중치라면 결과 동일).
- **파라미터**: `temperature=0.0`, `top_p=1.0`, `max_tokens=3500`(`LLM_MAXTOK`)
  - `temperature=0.0` 결정적 생성 — JSON 스키마 강제 없이 프롬프트만으로 출력 구조를 보장한다.
- **배치**: `LLM_BATCH=20` (후보를 20개씩 나눠 한 번에 평가)
- **후보 절단**: 각 후보 텍스트는 `cand_maxlen`(기본 280자)로 절단해 프롬프트에 넣는다. 특허 명세처럼 긴 후보는 이 값을 키워 사용.

## System prompt (`_SYS_DOC`)

```
당신은 기술사업화 매칭을 평가하는 전문가입니다. 기준이 되는 문서(기업·국가 R&D 과제·기술 수요·특허 등)와 후보 문서들 사이의 실질적 적합도를 냉정하게 평가합니다.
```

## User prompt

`기준 블록` + `"\n\n"` + `후보 목록` + `"\n\n"` + `가이드` 로 조립한다.

```
messages = [
  {"role": "system", "content": <_SYS_DOC>},
  {"role": "user",   "content": <기준 블록> + "\n\n" + <후보 목록> + "\n\n" + <가이드>},
]
```

### (a) 기준 블록

`{src_label}`에 기준측 유형 라벨이 들어간다.

```
[{src_label}]
{기준측 문서 전체 텍스트}
```

### (b) 후보 목록

`{cand_label}`에 후보측 유형 라벨이 들어간다. 배치당 최대 `LLM_BATCH`(20)개, 각 후보 텍스트는 `cand_maxlen`(기본 280)자로 절단.

```
[평가 대상 {cand_label} 목록]
{id}. {후보 문서 텍스트[:cand_maxlen]}
...
```

### (c) 가이드

`{src_label}`·`{cand_label}` 라벨이 첫 문장에 삽입된다.

```
각 [{cand_label}] 후보가 위 [{src_label}]와(과) 기술적으로 얼마나 적합·관련되는지 0~100 정수로 평가하세요.
  90-100: 핵심 기술/제품이 직접 부합하는 매우 높은 적합
  70-89 : 핵심 기술·응용 분야가 상당 부분 부합
  40-69 : 일부 요소만 관련(부분 적합)
  10-39 : 분야만 유사하거나 약하게 관련
  0-9   : 사실상 무관
키워드 표면 일치가 아니라 기술 내용·적용 관점의 실질 적합도를 보세요.
반드시 아래 JSON 형식으로만, 모든 후보에 대해 답하세요(reason 은 40자 이내 한국어):
{"results": [{"id": <후보번호>, "score": <0-100 정수>, "reason": "<근거>"}, ...]}
```

## 출력 / 파싱

- 기대 출력: `{"results": [{"id": <int>, "score": <0-100 int>, "reason": "<str>"}, ...]}`
- 파싱 절차:
  1. 응답에서 `{ ... "results" ... }` 패턴을 정규식으로 잘라내 `json.loads` 시도
  2. 파싱 실패 시 폴백: 개별 객체 단위 정규식으로 `id`·`score`·`reason`을 직접 추출
     - `\{\s*"id"\s*:\s*(\d+)\s*,\s*"score"\s*:\s*(\d+)\s*,\s*"reason"\s*:\s*"([^"]*)"`
  3. `score`는 `max(0, min(100, ...))`로 **0~100 클램프**, `reason`은 120자로 절단
  4. `{id: (score, reason)}` 형태로 반환

## 누락 처리

- 배치별로 평가하되, **응답에 빠진 후보**나 **LLM 호출 실패한 배치**의 후보는 기본값 `(0, "(LLM 응답 누락)")`으로 채운다.
- 즉 어떤 후보도 결과에서 사라지지 않으며, 평가되지 못한 후보는 `score=0`으로 하위 정렬된다.

## 활용 흐름 (권장)

1. 임베딩 유사도(코사인)·리랭커 등으로 후보를 1차 추림(비용이 큰 LLM 평가 대상 축소).
2. 이 프롬프트로 후보에 0~100 적합도 점수를 배치 평가.
3. 점수 내림차순 정렬 후 상위 N개 선정.
