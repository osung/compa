# 문서 키워드 추출 프롬프트 (범용)

**기업·과제·기술·특허·기술수요 등 임의의 기술사업화 문서**를 하나의 문자열로 받아, 검색·매칭에 유용한 핵심 기술 키워드를 추출한다.

이 문서는 특정 실행 파일이나 실행 하드웨어(NVIDIA GPU / Apple Silicon 등)에 의존하지 않는다. **Qwen 계열 LLM + 문서 문자열**만 있으면 그대로 재현할 수 있도록, 프롬프트 전문·파라미터·출력 파싱·규칙기반 폴백까지 자기완결적으로 기술한다.

## 개요

- **입력**: 제목/내용 구분 없이 문서 전체 텍스트 한 덩어리(문자열)
- **출력**: 키워드 문자열 리스트 (JSON `{"keywords": [...]}`) — **중요도가 높은 키워드부터 순서대로**
- **처리**: `System prompt` + (`문서 블록` + `가이드`) 를 LLM에 전달 → JSON 파싱 → (부족 시) 규칙기반 명사 추출로 보충

## 모델 / 파라미터

- **모델**: Qwen3.6-35B-A3B 계열 (35B MoE, 활성 3B, 4bit 양자화)
  - 실행 백엔드·하드웨어와 무관하게 동일 프롬프트로 동작한다(같은 가중치라면 결과 동일).
- **파라미터**: `temperature=0.0`, `top_p=1.0`, `max_tokens=600`
  - `temperature=0.0` 결정적 생성 — JSON 스키마 강제 없이 프롬프트만으로 출력 구조를 보장한다.
- **개수 제한**: `KW_MIN=10`(최소 목표), `KW_MAX=20`(최대·최종 절단)

## System prompt (`_KW_DOC_SYS`)

```
당신은 기술사업화 문서에서 핵심 기술 키워드를 추출하는 전문가입니다. 기업 소개·R&D 과제·기술 설명·특허 명세·기술 수요 등 어떤 유형의 문서가 들어오더라도, 문서의 기술적 실체를 대표하고 검색·매칭에 유용한 구체적 기술 용어만 선별합니다.
```

## User prompt

`문서 블록` + `"\n\n"` + `가이드(_KW_DOC_GUIDE)` 로 조립한다.

```
messages = [
  {"role": "system", "content": <_KW_DOC_SYS>},
  {"role": "user",   "content": <문서 블록> + "\n\n" + <_KW_DOC_GUIDE>},
]
```

### (a) 문서 블록

```
[문서]
{문서 전체 텍스트}
```

### (b) 가이드 (`_KW_DOC_GUIDE`)

`KW_MIN`(10)·`KW_MAX`(20) 값이 문자열 안에 삽입된다.


## 출력 / 파싱

- 기대 출력: `{"keywords": ["...", ...]}`
- 파싱 절차:
  1. 응답에서 `{ ... "keywords" ... }` 패턴을 정규식으로 잘라내 `json.loads` 시도
  2. 파싱 실패 시 폴백: `"keywords"` 배열 내부의 `"..."` 문자열들을 정규식으로 직접 추출
  3. 각 항목을 `strip`·따옴표 제거 후, **대소문자 무시 중복 제거**(LLM이 출력한 순서 = 중요도 순을 그대로 유지)
  4. 최대 `KW_MAX`(20)개로 절단 — 순서가 중요도 순이므로, 절단되는 것은 항상 **덜 중요한 뒤쪽 키워드**다

## 폴백: 규칙기반 명사 추출

LLM 결과가 `KW_MIN`개 미만이면(또는 LLM 호출 실패 시), 문서 텍스트에서 규칙기반으로 명사 키워드를 뽑아 이미 확보한 키워드 뒤에 이어 붙여 최대 `KW_MIN`개까지 채운다. LLM이 뽑은 키워드와 casefold 기준으로 중복되는 토큰은 추가하지 않는다.

이 규칙기반 정제의 목적은, 업종 문장체("반도체 제품 도매업", "생명공학에 관련된 시약")를 그대로 두면 행정/정책 문체로 흘러 매칭 품질이 떨어지므로 **핵심 명사만 남기는** 데 있다. 보충 키워드는 LLM이 뽑은 (중요도 순) 키워드 **뒤에** 문서 등장 순서대로 덧붙으므로, 전체 순서상 상대적으로 낮은 우선순위에 위치한다.

### 처리 절차

문서 텍스트를 다음 순서로 정제하여 토큰 리스트(등장 순서 유지)를 만든다.

1. **토큰화** — 한글/영문/숫자·밑줄을 제외한 구두점·구분자를 공백으로 치환한 뒤 공백 기준으로 어절 분리.
2. **1차 필터** — 다음 어절 제거
   - 길이 2글자 미만
   - **불용어(stopword)** 에 해당(아래 목록)
   - **연결어미·접속어** 에 해당(아래 목록)
3. **무의미 코드 제거** — 한글이 없는 ASCII 토큰 중
   - 순수 숫자(`030301`, `2024`)는 길이 무관 제거
   - 영문+숫자 혼합이면서 **5글자 이상**(`EH030301`, `RS232` 등, 과기표준분류 코드류)은 제거
   - 단, 4글자 이하 혼합(`5G`, `3D`, `4K`, `MP3`, `H264`)과 순수 영문 약어(`AI`, `IoT`, `RFID`)는 기술 용어로 보고 보존
4. **업종 접미사 제거** — 어절 끝의 업종 접미사를 떼고 남는 어간을 사용(어간이 접미사보다 길 때만).
   - 예) `반도체도매업 → 반도체`, `정밀화학공업 → 정밀화학`
   - 접미사: `판매업, 도매업, 소매업, 서비스업, 제조업, 생산업, 가공업, 임대업, 대행업, 무역업, 유통업, 공업, 업`
5. **어절 끝 조사 제거** — 남는 어간이 **2글자 이상**일 때만 조사를 뗀다(긴 조사부터 매칭).
   - 예) `생명공학에 → 생명공학`, `원료의약품의 → 원료의약품`
   - 조사: `으로서, 으로써, 에서의, 에게서, 에서, 에게, 으로, 와의, 과의, 에의, 에, 의, 을, 를`
   - 오제거(예: `결과→결`, `참가→참`)를 막기 위해 명사 말음과 충돌이 잦은 한 글자 조사(`가/이/은/는/도/만/로/와/과`)는 의도적으로 제외한다.
6. **불용어 동사 활용형 제거** — 어미를 뗀 어간이 불용어이면 토큰 전체를 버린다(일반 명사는 손상하지 않음).
   - 예) `이용한 → 이용`, `활용하여 → 활용`, `개발하는 → 개발` 이 모두 불용어이므로 제거
   - 검사 어미: `하여, 하는, 되어, 되는, 한, 해, 된, 되, 함, 됨`
7. **재검사** — 정제 후 남은 어간이 다시 2글자 미만·불용어·연결어이면 버린다. 통과한 것만 결과에 추가.

### 불용어(stopword)

위 **(b) 가이드의 불용어 목록과 동일**하다 — 하나의 불용어 목록을 LLM 가이드와 규칙기반 폴백이 함께 사용한다.

### 연결어미·접속어

```
관련된, 관련되, 위한, 통한, 대한, 관한, 따른, 인한,
위해, 통해, 위하여, 하는, 되는, 있는, 없는, 같은,
하고, 되고, 하며, 되며, 해서, 하여, 되어, 하면, 되면,
또는, 혹은, 그리고, 하지만, 그러나, 따라서, 그래서,
해당하는, 필요한, 가능한, 부대하는, 속하는, 부대되는
```

> 참고: 불용어 목록은 LLM 가이드와 규칙기반 폴백이 하나를 공유한다. 다만 쓰임은 다르다 — 가이드에서는 사람이 읽는 지시문의 일부로, 폴백에서는 토큰 매칭용 집합으로 쓰인다. '연결어미·접속어'는 폴백 전용 목록이다.
