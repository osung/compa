# COMPA 매칭 — 전체 프로세스 및 결과 정리

기업 진성수요 **78건**(담당자 6명, 익명 A~F)을 국가 R&D 과제 **약 96만 건**과 매칭하여
각 수요마다 적합도 높은 **Top5 과제 + 추천 근거**를 산출한 작업의 전체 과정과 검증 결과.

- 임베딩: `pro-sroberta` (SentenceTransformers)
- 어휘 검색: BM25 (scikit-learn 기반)
- 재랭커: `BAAI/bge-reranker-v2-m3` (크로스인코더)
- LLM: Qwen 3.x **35B-A3B** (Apple Silicon=MLX / CUDA=vLLM)

---

## 1. 최종 매칭 파이프라인 (실제 프로세스)

새 수요기술 1건이 들어오면 → Top5 까지:

1. **키워드 추출** — LLM이 수요기술명·내용·사양에서 핵심 기술 키워드 추출
2. **후보 검색(합집합)** — 임베딩 top1500 **∪** BM25 top1500 (리콜 확보)
3. **재랭킹** — 크로스인코더로 관련도순 정렬 → 상위 **200**
4. **LLM 정밀 채점** — 35B가 200개를 수요 사양 실제 충족도로 0~100 채점(표면 일치는 저점)
5. **Top5 선정** — 점수순 상위 5 (과제명 중복 제거)
6. **상세 근거 + 산출** — 4섹션 근거 생성, `xlsx`/`pkl` + 15컬럼 정리본 + 기업별 탭

- 구현: **`compa_match.py`** (통합 완료) / Colab: **`compa_colab.ipynb`**
- 무거운 단계는 4(LLM)뿐 → 3에서 top200으로 줄여 비용 고정.

## 2. 개선 과정 (왜 이 파이프라인인가)

### 2.1 문제 진단
- 초기 파이프라인 = **SBERT(임베딩) top100 → LLM**. **리콜 부족** — 관련 과제가 후보에조차
  못 오름(놓친 과제들의 임베딩 순위 중앙값 **약 9만 위**). 원인: 키워드↔과제 어휘 표현 격차.

### 2.2 검색 개선 실험 요약
| 방법 | 결과 / 판정 |
|---|---|
| top-N 확대 | 전건 포함에 top ≈ 60만(코퍼스 62%) 필요 → **폐기** |
| BM25(어휘) | 순위 9만 → **중앙 722위**로 급개선. 인덱스 31초, 검색 12ms |
| RRF 하이브리드 | 임베딩 순위가 나빠 희석 → **부적합** |
| BM25F(필드가중) | 소폭 개선(제목형↑ 설명문형↓), 전역가중 한계 |
| **크로스인코더 재랭킹** | 제목/설명문형 모두 회수, **최선** |
| HyDE(가상 키워드) | 키워드 공간에선 무효(잘 잡히던 것도 악화) → **폐기** |
| **합집합(임베딩∪BM25) + 재랭커** | **채택** — BM25 리콜 + 임베딩 정밀후보, 오매칭 완화 |

### 2.3 평가 전제 정정
- 참고용 "수동(시트) 매칭"은 **비전문가 선택 = 정답이 아님.** "수동 복구율"은 성능 지표로
  쓰지 않음(리콜 문제를 드러낸 참고용일 뿐).

---

## 3. 평가 방법 — 쌍대비교 (정답 없음)

- 정답 데이터가 없고, LLM 절대점수(0~100)는 배치 내 상대값이라 파이프라인 비교에 부적합.
- **쌍대비교(A/B)**: 같은 수요에 대해 [기존 top1] vs [신규 top1]을 35B가 "어느 쪽이 수요를 더
  잘 충족하나" 판정. **A/B 순서를 2회 뒤집어** 일관될 때만 승자 인정(위치편향 제거).
- 쌍대비교는 **평가 전용** — 실제 매칭 파이프라인에는 포함되지 않음.

---

## 4. 검증 결과 — 합집합 vs 기존(SBERT) (전체 78수요)

35B 쌍대판정(순서 2회) 결과:

| 담당자 | 신규(합집합) 승 | 기존 승 | 무승부 | 승률(무승부 제외) |
|:---:|:---:|:---:|:---:|:---:|
| A | 13 | 1 | 1 | 93% |
| B | 11 | 0 | 4 | 100% |
| C | 14 | 0 | 1 | 100% |
| D | 9 | 2 | 4 | 82% |
| E | 11 | 1 | 3 | 92% |
| F | 2 | 0 | 1 | 100% |
| **합계** | **60** | **4** | **14** | **94%** |

- **신규(합집합) 60승 · 기존 4승 · 무승부 14** → **무승부 제외 승률 94%** (전체 기준 77%).
- 전 담당자에서 합집합 우세. 기존이 이긴 건 78건 중 **단 4건**(기존도 우수했던 소수 케이스).
- 무승부 14건은 대체로 양쪽 모두 우수(순서편향으로 우열 불명).

**→ 정답 없이도 쌍대비교로 합집합 파이프라인의 우수성이 정량 확인됨.**

## 5. 성능 / 비용 (실측)

- **매칭 속도**: 수요 1건당 ~4.8분 (재랭킹 ~80초 + 35B 채점(top200) + 상세근거). 78건 전체 ~5시간.
- **재랭킹 작업량**: 합집합 평균 **2,933쌍/수요** (임베딩·BM25 후보 겹침 2%뿐).
  → BM25 top3000 대비 **−2%**로 사실상 동일. 즉 임베딩 합류의 추가 비용은 거의 없음(~ms).
- **속도 대비**: BM25-only 대비 손해 거의 없음. 기존 SBERT 대비 ~1.5배(재랭킹 추가분).
- **메모리**: 단일 프로세스 피크 ~37GB(128GB 중) — 안전.

### 5.1 LLM 모델 비교 — 27B(dense) vs 35B-A3B(MoE)
서수민 15수요를 두 모델로 매칭(동일 키워드·후보풀)하고 비교:

- **속도**: 35B-A3B는 MoE(활성 3B)라 로컬(MLX) ~5분/건. **27B는 dense라 ~28분/건**(전체 ~30h로 비현실적).
- **품질(제3자 판정, Claude)**: 35B 6승 · 27B 5승 · 무 4 → **사실상 대등**(35B 근소 우위).
  - 참고: 27B가 스스로 판정하면 27B 71% 승 → **자기편향**. 공정 판정은 제3자로 해야 함.
  - Top5 겹침 56%, 1위 동일 20% → 모델 바꾸면 상위 선택이 꽤 갈리나 품질은 동급.
- **결론**: **로컬 운영은 35B-A3B 유지**(품질 대등 + 훨씬 빠름). 27B가 필요하면 Colab/vLLM(FP8).
